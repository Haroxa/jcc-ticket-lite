# 13 页面原型审查评估

## 1. 审查视角

本审查站在日常使用者角度，而不是开发者角度。

假设使用者包括：

- 管理员：需要看全局数据、导入导出、作废恢复、维护数据。
- 录入员：高频新增存入和取用记录。
- 查看者：主要查看余额、个人历史和流水。

审查重点：

- 是否容易上手。
- 是否适合电脑、平板、手机操作。
- 高频操作是否足够顺。
- 列表查询和分页是否够用。
- 页面信息是否稳定、不迷路。
- 后续正式实现是否容易扩展。

## 2. 总体评价

当前原型已经从“页面骨架”进入了“可试用流程原型”阶段。

做得比较好的地方：

- 已有登录入口，符合真实系统使用路径。
- 首页、快速录入、存票人、记录、个人历史、导入导出、系统信息等主要页面齐全。
- 快速录入路径比较明确。
- 计算工具放到快速录入下方是合理的，减少了页面跳转。
- 存票人和存取记录已经有分页和顺序编号。
- 记录作废/恢复已有交互雏形。
- 多端布局方向基本正确：电脑侧边栏、平板顶部导航、手机底部导航。

当前主要问题：

- 导航、分页、筛选和历史查询还停留在“能用”，没有达到“好用”。
- 多端适配已经有框架，但细节还需要继续压实。
- 个人历史页面能力偏弱，和正式业务价值不匹配。
- 数据量变大后的列表操作效率还不够。
- 状态反馈、危险操作确认、空状态和异常状态不够完整。

## 3. 高优先级问题

### P0-1 电脑端侧边栏没有固定

问题：

电脑端左侧导航目前随页面整体滚动。对于记录列表、个人历史、导入预览这类长页面，用户滚动到下面后会失去导航入口。

影响：

- 用户容易迷路。
- 长页面切换模块不方便。
- 后续数据量变大后体验会明显下降。

建议：

- 电脑端侧边栏使用固定或粘性布局。
- 顶部栏也建议保持粘性，至少在电脑端固定。
- 主内容区独立滚动，导航区不跟随滚走。

建议布局：

```text
顶部栏 sticky
左侧导航 sticky / fixed
右侧主内容滚动
```

### P0-2 个人历史没有分页

问题：

个人历史页当前只是直接展示某个存票人的所有记录，没有分页，也没有更完整的排序和筛选状态。

影响：

- 高频存票人历史记录多时页面会很长。
- 手机端查看困难。
- 无法快速跳到较早记录。

建议：

- 个人历史增加分页。
- 显示顺序编号。
- 支持按日期倒序/正序切换。
- 支持只看存入、只看取用、只看作废。
- 分页状态展示总条数、当前页、总页数。

### P0-3 分页组件不够完善

问题：

当前分页只有上一页、下一页、当前页和总数，不支持直接跳页，也不能调整每页条数。

影响：

- 数据多时跳转效率低。
- 管理员查历史记录时需要反复点下一页。
- 不同设备上用户期望不同：电脑希望更多行，手机希望少量卡片。

建议分页组件包含：

- 上一页
- 下一页
- 首页
- 末页
- 当前页码
- 可输入页码跳转
- 每页条数选择：10 / 20 / 50
- 总条数

建议显示：

```text
共 128 条  每页 [20 v]  首页 上一页  第 [3] / 7 页  下一页  末页
```

手机端可以简化为：

```text
上一页  第 3 / 7 页  下一页
每页 10 条
```

### P0-4 记录列表筛选缺少“重置”和“明确查询状态”

问题：

筛选框视觉已经比之前好，但还缺少两个关键能力：

- 一键重置筛选。
- 当前筛选结果说明。

影响：

- 用户筛选后可能忘记当前列表不是全部数据。
- 查不到数据时不知道是没记录，还是筛选条件太窄。

建议：

- 筛选区增加 `查询` 和 `重置` 按钮。
- 列表上方显示当前结果摘要。

示例：

```text
当前显示：正常记录 / 存入 / 关键词“张三”，共 12 条
```

## 4. 中优先级问题

### P1-1 快速录入页还可以更贴近高频操作

问题：

快速录入已经可用，但还可以进一步减少操作成本。

建议：

- 存票人搜索框替代普通下拉框。
- 搜索结果显示余额，方便选人。
- 保存成功后保留存票人和类型，光标回到票数。
- 提供“保存并继续录入”状态反馈。
- 对取用类型更明显地显示当前余额和取用后余额。

示例：

```text
当前余额：1200
本次取用：300
取用后余额：900
```

### P1-2 作废/恢复操作缺少确认弹窗

问题：

当前原型点击作废/恢复会直接切换状态。

影响：

- 真实使用中容易误点。
- 作废原因没有被强制填写。

建议：

- 作废必须弹窗确认。
- 作废原因必填。
- 恢复也需要二次确认。
- 弹窗展示被操作记录的摘要。

### P1-3 首页汇总缺少可追溯入口

问题：

首页有余额排行和最近记录，但最近记录不能直接定位到具体记录。

建议：

- 最近记录点击后跳转到记录列表并高亮该记录。
- 余额排行点击进入个人历史，这点已有雏形，应保留。
- 今日存入/今日取用统计卡点击后进入记录列表并自动带筛选。

### P1-4 移动端“更多”页还没有明确内容

问题：

手机底部导航的 `更多` 当前直接跳到导入导出，不是真正的更多菜单。

影响：

- 手机端找系统信息、导出、退出等入口不直观。

建议：

新增一个手机端 `更多` 页面或菜单，包含：

- 个人历史
- 导入导出
- 系统信息
- 退出登录

计算工具不需要放入更多，因为它已经在快速录入下方。

### P1-5 系统信息页方向正确，但维护入口需要排序

问题：

系统信息页比原来的设置页更合理，但维护入口还比较松散。

建议维护入口按使用频率排序：

1. 导出完整备份
2. 查看操作日志
3. 检查异常数据
4. 导入迁移数据
5. 初始化管理员说明

## 5. 低优先级问题

### P2-1 表格密度和卡片密度还可以继续优化

问题：

电脑端表格行距偏舒适，适合原型，但正式工具可能需要更高信息密度。

建议：

- 电脑端提供紧凑列表。
- 手机端保持卡片布局。
- 平板端使用简化表格或卡片混合。

### P2-2 缺少空状态和错误状态设计

需要补充：

- 无存票人。
- 无记录。
- 筛选无结果。
- 登录失败。
- 导入失败。
- 保存失败。
- 网络异常。

### P2-3 缺少加载中状态

正式接入 Cloudflare API 后，需要：

- 页面加载骨架。
- 保存按钮 loading。
- 导入预览 loading。
- 分页切换 loading。

### P2-4 缺少权限差异展示

不同角色看到的按钮应该不同：

- 查看者不应看到保存、作废、恢复、导入按钮。
- 录入员是否能作废需要明确。
- 管理员才看到系统维护入口。

## 6. 页面逐项评估

### 登录页

当前评价：

- 有入口，满足原型需要。
- 视觉简单直接。

建议：

- 增加登录失败状态。
- 增加“记住登录状态”的设计说明。
- 移动端确认输入框和按钮高度足够。

### 首页汇总

当前评价：

- 信息结构清楚。
- 余额排行和最近记录是正确方向。

建议：

- 统计卡可点击进入对应筛选结果。
- 最近记录支持点击定位。
- 增加最后更新时间。

### 快速录入

当前评价：

- 主流程可用。
- 计算工具放在下方合理。

建议：

- 存票人选择从下拉升级为搜索选择。
- 增加取用后余额预览。
- 保存成功后光标回到票数。
- 计算工具可以折叠，避免页面过长。

### 存票人管理

当前评价：

- 已有编号、分页、搜索和状态筛选。

建议：

- 增加每页条数调整。
- 增加编辑入口。
- 增加余额排序。
- 增加“只看有余额”筛选。

### 存取记录

当前评价：

- 已有编号、分页、筛选、作废恢复。

建议：

- 增加日期范围筛选。
- 增加页码跳转和每页条数。
- 作废恢复改为弹窗确认。
- 支持按日期、票数排序。

### 个人历史

当前评价：

- 作为查看页存在，但能力不足。

建议：

- 增加分页。
- 增加编号。
- 增加类型、状态筛选。
- 增加日期范围筛选结果摘要。
- 支持导出当前个人记录。

### 导入导出

当前评价：

- 入口存在，但交互还偏概念。

建议：

- 导入预览表也需要分页。
- 错误行需要展开详情。
- 导入前要显示影响范围：新增多少存票人、新增多少记录、跳过多少重复。

### 系统信息

当前评价：

- 比复杂设置页更适合第一版。

建议：

- 增加当前环境、数据库连接状态、最近备份时间。
- 维护入口按优先级排序。

## 7. 下一版原型调整建议

建议下一轮优先做这些：

1. 固定电脑端侧边栏和顶部栏。
2. 抽象一个统一分页组件，用到存票人、存取记录、个人历史、导入预览。
3. 分页组件支持页码输入和每页条数调整。
4. 个人历史增加编号、分页、筛选。
5. 记录列表增加日期范围筛选、查询/重置按钮、筛选摘要。
6. 快速录入的存票人选择改成搜索选择。
7. 作废和恢复增加确认弹窗。
8. 手机端增加真正的更多菜单。

## 8. 结论

当前原型方向是对的：它已经能表达一个轻量 Cloudflare 存票系统的主要页面和核心流程。

但如果按真实使用者标准，现在还不能算“顺手”。最需要补的是列表型页面的成熟度：固定导航、完整分页、个人历史分页、筛选状态说明、危险操作确认。

下一版原型应重点解决“数据多起来以后还能不能快查、快录、快改”的问题。
